# ADR-002: PhenoRuntime Security Hardening

## Status

**Accepted**

## Context

Container security is critical as containers often run with significant privileges and access to sensitive data. Defense in depth requires multiple security layers from kernel to application.

Key challenges:

1. **Privilege Escalation**: Preventing container escape
2. **Image Security**: Ensuring only trusted images run
3. **Runtime Security**: Detecting anomalous behavior
4. **Network Isolation**: Containing lateral movement
5. **Secret Management**: Secure credential handling
6. **Compliance**: Meeting security standards

## Decision

We will implement comprehensive security hardening:

### 1. Container Isolation

**Rationale**: Multiple isolation layers prevent escape.

**Decision Details**:
- Namespaces for process, network, mount isolation
- cgroups for resource limits
- Seccomp profiles for syscall filtering
- AppArmor/SELinux for mandatory access control
- Capabilities dropping (remove all, add only needed)
- Read-only root filesystem
- No new privileges flag

### 2. Image Security Pipeline

**Rationale**: Trust starts with the image.

**Decision Details**:
- Base image from trusted sources only
- Image signing with cosign/Sigstore
- Vulnerability scanning in CI/CD
- Image provenance tracking
- Reject images with critical vulnerabilities
- Regular base image updates

### 3. Runtime Security Monitoring

**Rationale**: Detection is essential for security.

**Decision Details**:
- Falco for runtime threat detection
- System call monitoring
- File integrity monitoring
- Network anomaly detection
- Container drift detection
- Integration with SIEM

### 4. Network Security

**Rationale**: Network is the attack vector.

**Decision Details**:
- Default deny network policies
- Service mesh for mTLS
- Egress filtering
- DNS policy enforcement
- Network segmentation by sensitivity
- API gateway for external access

### 5. Secret Management

**Rationale**: Secrets in containers are a vulnerability.

**Decision Details**:
- No secrets in images
- Runtime secret injection
- Integration with Vault, AWS Secrets Manager
- Ephemeral secret mounts
- Automatic secret rotation
- Audit logging of secret access

## Consequences

### Positive

1. **Protection**: Multiple defense layers
2. **Detection**: Runtime threat detection
3. **Compliance**: Meets security standards
4. **Trust**: Verifiable image provenance
5. **Automation**: Security as code
6. **Visibility**: Security monitoring and audit

### Negative

1. **Complexity**: Many security layers to manage
2. **Overhead**: Security monitoring adds overhead
3. **False Positives**: Runtime detection may alert on legitimate activity
4. **Maintenance**: Security profiles need updates
5. **Learning Curve**: Security concepts require training

### Neutral

1. **Performance**: Security vs performance tradeoffs
2. **Tools**: Multiple security tools to integrate
3. **Policy**: Security policy governance required

## Alternatives

### Alternative 1: Host Security Only

**Considered**: Rely on host security, minimal container security.

**Rejected Because**:
- Container escape would compromise host
- No runtime detection
- Insufficient for compliance

### Alternative 2: VM Isolation Only

**Considered**: Use VMs for all isolation.

**Rejected Because**:
- Higher overhead
- Slower provisioning
- Less efficient resource usage

### Alternative 3: No Runtime Security

**Considered**: Only image scanning, no runtime monitoring.

**Rejected Because**:
- Cannot detect runtime attacks
- No response capability
- Insufficient defense in depth

### Alternative 4: Manual Security

**Considered**: Manual security reviews instead of automation.

**Rejected Because**:
- Doesn't scale
- Inconsistent application
- Slow feedback loop
