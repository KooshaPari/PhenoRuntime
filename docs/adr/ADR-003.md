# ADR-003: PhenoRuntime Resource Scheduling

## Status

**Accepted**

## Context

Efficient resource scheduling is essential for maximizing hardware utilization while ensuring application performance. The scheduler must balance resource requests, priorities, and constraints across diverse workloads.

Key challenges:

1. **Fairness**: Sharing resources across tenants
2. **Efficiency**: High cluster utilization
3. **Latency**: Fast scheduling decisions
4. **Constraints**: Hardware, topology, and policy constraints
5. **Preemption**: Handling resource contention
6. **Autoscaling**: Dynamic resource adjustment

## Decision

We will implement a multi-level scheduling architecture:

### 1. Scheduling Framework

**Rationale**: Pluggable scheduling enables optimization.

**Decision Details**:
- Kubernetes scheduler as primary
- Custom scheduler profiles for different workload types
- Scheduling framework plugins
- Preemption and priority support
- Batch scheduling for high-throughput
- Gang scheduling for distributed jobs

### 2. Resource Quotas and Limits

**Rationale**: Controls prevent resource exhaustion.

**Decision Details**:
- ResourceQuota per namespace
- LimitRange for default constraints
- PodDisruptionBudget for availability
- CPU request/limit with burst support
- Memory limit with OOM handling
- Ephemeral storage limits

### 3. Quality of Service

**Rationale**: Different workloads need different QoS.

**Decision Details**:
- **Guaranteed**: Requests == Limits (critical workloads)
- **Burstable**: Requests < Limits (normal workloads)
- **BestEffort**: No requests/limits (batch/low priority)
- QoS affects eviction priority
- QoS affects CPU shares
- Resource quota by QoS class

### 4. Autoscaling

**Rationale**: Dynamic adjustment improves efficiency.

**Decision Details**:
- Horizontal Pod Autoscaler (HPA) for replicas
- Vertical Pod Autoscaler (VPA) for resources
- Cluster Autoscaler for nodes
- Custom metrics support
- Predictive scaling based on schedule
- Scale-down delays for stability

### 5. Topology Awareness

**Rationale**: Hardware topology affects performance.

**Decision Details**:
- NUMA topology awareness
- GPU topology for multi-GPU workloads
- Network topology for data locality
- Zone/region spread for availability
- Pod affinity/anti-affinity rules
- Topology spread constraints

## Consequences

### Positive

1. **Efficiency**: Better resource utilization
2. **Performance**: Topology-aware placement
3. **Fairness**: Resource quotas prevent starvation
4. **Flexibility**: Multiple scheduling strategies
5. **Automation**: Autoscaling reduces manual work
6. **Reliability**: Preemption and eviction handling

### Negative

1. **Complexity**: Sophisticated scheduling algorithms
2. **Latency**: Complex scheduling takes time
3. **Tuning**: Requires expertise to configure
4. **Predictability**: Autoscaling can surprise
5. **Overcommit**: Risk of resource exhaustion

### Neutral

1. **Capacity Planning**: Still needed for base capacity
2. **Monitoring**: Extensive monitoring required
3. **Policies**: Many scheduling policies to manage

## Alternatives

### Alternative 1: Static Allocation

**Considered**: Fixed resource allocation per workload.

**Rejected Because**:
- Poor utilization
- Manual adjustment burden
- Cannot adapt to demand

### Alternative 2: Simple Round-Robin

**Considered**: Simple scheduling without optimization.

**Rejected Because**:
- Ignores resource requirements
- No topology awareness
- Poor performance

### Alternative 3: External Scheduler

**Considered**: Use only external scheduler (e.g., Mesos).

**Rejected Because**:
- Additional complexity
- Integration challenges
- Duplicates Kubernetes capabilities

### Alternative 4: Manual Scheduling

**Considered**: Users manually specify placement.

**Rejected Because**:
- Doesn't scale
- Requires deep cluster knowledge
- Error-prone
