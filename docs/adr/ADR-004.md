# ADR-004: PhenoRuntime Developer Experience

## Status

**Accepted**

## Context

Developer productivity is critical for platform adoption. The runtime must provide simple workflows for local development, testing, and deployment while maintaining production parity.

Key challenges:

1. **Local Development**: Replicating production locally
2. **Hot Reload**: Fast iteration cycles
3. **Debugging**: Container debugging capabilities
4. **Testing**: Integration and end-to-end testing
5. **Tooling**: IDE and CLI integration
6. **Onboarding**: New developer ramp-up time

## Decision

We will prioritize developer experience with the following:

### 1. Local Development Environment

**Rationale**: Development should feel native, not foreign.

**Decision Details**:
- `pheno dev up` for local environment
- Docker Compose integration
- File watching and auto-restart
- Service discovery for local services
- Shared development databases
- Environment parity with production

### 2. Hot Reload and Live Sync

**Rationale**: Fast feedback loops increase productivity.

**Decision Details**:
- File sync from host to container
- Framework-specific hot reload support
- Incremental builds in containers
- Selective sync (ignore patterns)
- Bi-directional sync where needed
- Conflict resolution strategies

### 3. Debugging Support

**Rationale**: Debugging containers must be straightforward.

**Decision Details**:
- IDE debugging integration (VSCode, GoLand, etc.)
- Remote debugger port forwarding
- Delve, pdb, node-inspector support
- Log tailing and filtering
- Container shell access
- Core dump capture

### 4. Testing Integration

**Rationale**: Testing should work seamlessly with runtime.

**Decision Details**:
- Testcontainers integration
- Isolated test databases
- Test data seeding
- Parallel test execution
- CI/CD pipeline integration
- Coverage reporting

### 5. CLI and Tooling

**Rationale**: Command-line experience affects productivity.

**Decision Details**:
- `pheno` CLI for all operations
- Shell completion for bash/zsh/fish
- Progress indicators and rich output
- Consistent command structure
- Plugin architecture for extensibility
- Clear error messages with suggestions

## Consequences

### Positive

1. **Productivity**: Faster development cycles
2. **Adoption**: Good DX drives platform adoption
3. **Quality**: Easy testing leads to better quality
4. **Debugging**: Faster bug resolution
5. **Onboarding**: New developers productive quickly
6. **Satisfaction**: Happy developers

### Negative

1. **Complexity**: Dev tooling adds complexity
2. **Maintenance**: Tools need ongoing maintenance
3. **Resources**: Local environments resource-intensive
4. **Divergence**: Risk of dev/prod differences
5. **Learning**: New tools to learn

### Neutral

1. **Performance**: Local environment performance varies
2. **Disk Space**: Containers consume disk
3. **Updates**: Frequent tooling updates

## Alternatives

### Alternative 1: Remote Development Only

**Considered**: Only support cloud development environments.

**Rejected Because**:
- Network dependency
- Latency issues
- Internet required
- Less control

### Alternative 2: Native Development

**Considered**: No containers for local development.

**Rejected Because**:
- Dev/prod divergence
- "Works on my machine"
- Dependency conflicts

### Alternative 3: VM-Based

**Considered**: Use VMs instead of containers for dev.

**Rejected Because**:
- Slower startup
- Higher overhead
- More complex setup

### Alternative 4: No Tooling

**Considered**: Raw Docker commands only.

**Rejected Because**:
- Steep learning curve
- Inconsistent workflows
- Error-prone
