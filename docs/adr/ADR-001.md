# ADR-001: PhenoRuntime - Container Runtime Architecture

## Status

**Accepted**

## Context

The Phenotype ecosystem requires a robust container runtime for executing workloads across development, testing, and production environments. The runtime must support multiple container technologies while providing consistent interfaces and management capabilities.

Key challenges:

1. **Multi-Platform**: Support Linux, macOS, and Windows
2. **Security**: Sandboxed execution with minimal privileges
3. **Performance**: Low overhead container startup
4. **Orchestration**: Integration with Kubernetes and other schedulers
5. **Developer Experience**: Simple local development workflows
6. **Resource Management**: CPU, memory, and I/O controls

## Decision

We will implement a pluggable container runtime architecture:

### 1. Runtime Abstraction Layer

**Rationale**: Unified interface enables multiple backend support.

**Decision Details**:
- `Runtime` trait defining common operations
- Pluggable runtime implementations
- Runtime selection at configuration time
- Health checking and monitoring hooks
- Graceful degradation if runtime unavailable

### 2. Supported Runtimes

**Rationale**: Different environments require different runtimes.

**Decision Details**:
- **containerd**: Production Linux runtime
- **podman**: Rootless, daemonless alternative
- **Docker**: Widely compatible, developer-friendly
- **gVisor**: Sandboxed, security-focused
- **Kata Containers**: VM-level isolation
- **WSL2**: Windows Linux compatibility
- **colima/Lima**: macOS container support

### 3. Image Management

**Rationale**: Efficient image handling is critical for performance.

**Decision Details**:
- OCI-compliant image support
- Layer caching and deduplication
- Pull-through caching registry
- Image signing and verification (cosign)
- Private registry authentication
- Image scanning integration

### 4. Networking Model

**Rationale**: Container networking must be flexible and secure.

**Decision Details**:
- CNI (Container Network Interface) plugins
- Bridge networking for local development
- Overlay networking for clusters
- Host networking option for performance
- Custom DNS resolution
- Network policies for isolation

### 5. Storage Architecture

**Rationale**: Persistent and ephemeral storage needs differ.

**Decision Details**:
- Overlay2 for efficient image layers
- Volume mounts for persistent data
- tmpfs for high-speed temporary storage
- Bind mounts for development
- Snapshot support for state management
- Storage quota enforcement

## Consequences

### Positive

1. **Flexibility**: Multiple runtime options for different needs
2. **Security**: Sandboxed options for untrusted workloads
3. **Compatibility**: OCI standard ensures portability
4. **Performance**: Choose appropriate runtime for workload
5. **Developer Experience**: Local development simplified
6. **Production Ready**: Enterprise runtime options

### Negative

1. **Complexity**: Supporting multiple runtimes adds complexity
2. **Testing**: Must test against all supported runtimes
3. **Documentation**: More options to document
4. **Configuration**: Runtime selection requires understanding
5. **Inconsistency**: Subtle differences between runtimes

### Neutral

1. **Resource Usage**: Different runtimes have different footprints
2. **Updates**: Runtime updates must be coordinated
3. **Support**: Multiple runtime versions to support

## Alternatives

### Alternative 1: Single Runtime

**Considered**: Only support containerd.

**Rejected Because**:
- Poor developer experience for local development
- macOS/Windows support problematic
- Too rigid for diverse use cases

### Alternative 2: Docker Only

**Considered**: Standardize on Docker everywhere.

**Rejected Because**:
- Licensing concerns
- Security model not suitable for all cases
- Daemon requirement problematic

### Alternative 3: VM Only

**Considered**: Use VMs instead of containers.

**Rejected Because**:
- Higher overhead
- Slower startup
- Less resource efficient

### Alternative 4: No Abstraction

**Considered**: Direct runtime API calls.

**Rejected Because**:
- Vendor lock-in
- Hard to switch runtimes
- No unified management
